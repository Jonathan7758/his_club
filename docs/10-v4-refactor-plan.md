# 10 — v4.0 重构方案：历史公众号系列分析引擎

> **创建日期**: 2026-05-09
> **目标阶段**: v4.0 — 从"内容生成工作流"重构为"TG驱动的系列分析引擎"
> **开发模式**: 严格TDD (测试先行 → 实现 → 验证)
> **指导文件**: 本文档为 v4.0 开发的唯一权威设计文件

---

## 1. 重构动机

v3.0 已完成"热点发现→多角度分析→公众号内容生成"的完整管道，但运营中发现：

1. **公众号生成不是核心瓶颈** — 实际内容输入来自运营团队的深度分析材料
2. **真正的价值是分析** — 系列内容的选题评估、角度设计、子主题拆分
3. **交互方式需适配运营** — TG 命令行模式更适合快速/高频的分析请求

因此 v4.0 去掉生成层，将核心聚焦于"接收分析输入→多维分析评分→系列设计导出"。

## 4. 数据流（端到端）

```
TG用户 /analysis
    │
    ▼
tg_bot.py → session_mgr 创建session (状态=WAITING_CONTENT)
    │
    ├─ 用户逐条发送内容 (自动拆分超长消息)
    │  session累积拼接
    │
    │ 用户发送"开始分析" → 状态=SUMMARIZING
    ▼
session_mgr → LLM提取: 主主题 + 关键论点列表 → 返回TG用户确认
    │
    │ 用户确认 → 状态=ANALYZING
    ▼
┌─ 外部数据源层 ──────────────────────────┐
│ hotspot_scanner.py → NewsNow当前热度     │
│ engines.py         → 搜狗微信/网页/豆瓣  │
│ douban.py          → 豆瓣书评/小组       │
└─────────────────────────────────────────┘
    │
┌─ 内部辩论分析层 ────────────────────────┐
│ forum_engine.py    → 7维Agent辩论       │
│ mirofish.py        → 7方利益博弈推演    │
│ graph_analyzer.py  → 知识图谱盲区/聚类  │
│ fact_checker.py    → 事实断言验证       │
└─────────────────────────────────────────┘
    │
    ▼
scorer.py → 3维评分(热度/独特度/传播) + 每项附证据 → TG用户确认
    │
    │ 用户确认 → 状态=DESIGNING
    ▼
series_designer.py → 主主题按7维拆分为子系列
  每个子系列:
    - 子主题名称 (公众号化命名)
    - 核心观点 (1句话提炼)
    - 内容大纲 (3-5段标题目录)
    - 金句 (2-3句传播向金句)
  → 返回TG用户确认/调整
    │
    │ 用户确认 → 状态=SUB_SCORING
    ▼
scorer.py → 对每个子主题独立评分 (复用数据源+分析层)
    │
    ▼
doc_exporter.py → 汇总生成MD系列设计文档 → 返回TG下载
```

### 5.5 数据库新增表

```sql
CREATE TABLE IF NOT EXISTS analysis_sessions (
    id          VARCHAR(16) PRIMARY KEY,
    tg_chat_id  BIGINT NOT NULL,
    status      VARCHAR(20) DEFAULT 'WAITING_CONTENT',  
    -- WAITING_CONTENT | COLLECTING | SUMMARIZING | ANALYZING 
    -- | CONFIRM_SCORE | DESIGNING | CONFIRM_DESIGN | SUB_SCORING | COMPLETED
    raw_content TEXT,          -- 用户输入的全部原文
    main_topic  TEXT,          -- LLM提取的主主题
    key_points  JSONB,         -- LLM提取的关键论点列表
    analysis    JSONB,         -- 完整分析结果
    scores      JSONB,         -- 评分结果
    sub_series  JSONB,         -- 子系列列表
    sub_scores  JSONB,         -- 子系列评分
    doc_md      TEXT,          -- 最终MD文档
    created_at  TIMESTAMPTZ DEFAULT NOW(),
    updated_at  TIMESTAMPTZ DEFAULT NOW()
);

CREATE TABLE IF NOT EXISTS series_designs (
    id            VARCHAR(16) PRIMARY KEY,
    session_id    VARCHAR(16) REFERENCES analysis_sessions(id),
    main_topic    TEXT NOT NULL,
    sub_topics    JSONB NOT NULL,   -- [{name, viewpoint, outline[], quotes[], score}]
    full_doc_md   TEXT,
    full_doc_json JSONB,
    exported_at   TIMESTAMPTZ DEFAULT NOW()
);
```

## 6. 评分维度详细设计

| 维度 | 权重 | 数据来源 | 证据类型 | 满分 |
|------|------|----------|----------|------|
| **互联网热度** | 0.30 | NewsNow API 近7天热点 + 搜狗网页结果数 + 搜狗微信文章数 | 结果数量、跨平台提及数、相关文章趋势 | 10 |
| **角度独特度** | 0.40 | 搜狗微信相似度检测 + ForumEngine 7维盲区 + 知识图谱Gap | 竞品覆盖百分比、未覆盖维度数量、KG盲区数量 | 10 |
| **预测传播热度** | 0.30 | MiroFish博弈冲突度 + 情感标签强度 + 7维交叉批评争议度 | 博弈冲突激烈度、争议点数量、情感标签(猎奇/争议/共情)强度 | 10 |

**综合评分公式**:
```
total_score = 热度 × 0.30 + 独特度 × 0.40 + 传播热度 × 0.30
```

## 7. 子系列拆分逻辑

```
输入: 主主题 + 7维分析结果 + 外部数据
    ↓
LLM按7维框架拆分:
  ┌─ 政治制度 → 子主题1: 聚焦权力结构/制度变迁/政治博弈
  ├─ 经济财政 → 子主题2: 聚焦经济数据/财政危机/商业势力
  ├─ 军事战略 → 子主题3: 聚焦战役复盘/军事技术/地缘博弈
  ├─ 文化社会 → 子主题4: 聚焦文化冲击/社会阶层/思潮演变
  ├─ 关键人物 → 子主题5: 聚焦人物群像/决策逻辑/人性闪光
  ├─ 技术演进 → 子主题6: 聚焦技术变革/生产力/发明传播
  └─ 地理环境 → 子主题7: 聚焦地理棋局/环境约束/空间博弈
    ↓
每个子主题生成:
  - name: 子主题名称 (公众号化命名，含吸引力)
  - viewpoint: 核心观点 (1句话浓缩)
  - outline: 内容大纲 (3-5段标题目录)
  - quotes: 金句 (2-3句传播向金句)
    ↓
用户确认/调整 → 每个子主题独立评分 → 汇总输出
```

```
  ├── 重构: api.py (裁剪端点, 新增TG webhook)
  ├── 更新: database.py (新增2表)
  └── tests/test_api.py                 → RED → GREEN → REFACTOR

Phase 6: 集成测试
  ├── tests/test_integration.py         → 全流程端到端
  └── tests/test_e2e.py                 → TG bot模拟交互
```

### 9.3 TDD 纪律

每步严格执行 RED → GREEN → REFACTOR：

1. **RED**: 先写失败测试（禁止跳过）
2. **GREEN**: 写最少代码使测试通过
3. **REFACTOR**: 在绿色状态下重构，保持全绿

完整工作流：`pytest tests/test_xxx.py -v --cov=src/xxx --cov-report=term-missing`
