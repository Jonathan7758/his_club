## 2. 部署拓扑（两套方案）

### 方案 A：香港 + 马来双区（原方案）

```
用户/运营
    │
    ▼
┌─────────────────┐     VPN/专线     ┌─────────────────┐
│   香港节点 (HK)  │◄───────────────▶│  马来节点 (MY)   │
│                 │                 │                 │
│ • API Gateway   │                 │ • 爬虫节点       │
│ • Flask Web UI  │                 │ • MindSpider     │
│ • LLM 调用代理  │                 │ • PostgreSQL     │
│ • ReportEngine  │                 │ • Playwright     │
│ • ForumEngine   │                 │ • 代理IP池       │
│ • MiroFish Lite │                 │                 │
│                 │                 │                 │
│ LLM API (国内)  │                 │ 社交媒体平台     │
└─────────────────┘                 └─────────────────┘
```

**分工逻辑**：
- **香港节点**：计算密集型（LLM调用、分析、生成），国内API延迟低
- **马来节点**：IO密集型（爬虫、代理IP池），访问国内外社媒自由度高

### 方案 B：火山云单机部署 — 实际配置

```
火山引擎北京 4C8G (单机全服务)
│
├── PostgreSQL (Docker)        1.5GB   ← 舆情数据仓库
├── Flask Web UI (port 5000)   1.5GB   ← 运营看板+API
├── ForumEngine (7维辩论)              ← LLM调用为主
├── QueryEngine / MediaEngine          ← LLM调用为主  
├── InsightEngine                      ← LLM调用为主
├── ReportEngine                       ← LLM调用为主
├── MiroFish Lite (V1.0)       1.0GB   ← 历史推演，月4次错峰
├── MindSpider + Playwright    1.5GB   ← 日扫描10分钟，凌晨触发
├── Redis (Docker)             0.2GB   ← 缓存+任务队列
├── Nginx + APScheduler        0.1GB   ← 反向代理+定时任务
└── 系统预留                    ~1.3GB
                                ─────
峰值内存 ~6.0GB / 8GB ✅

备机:
├── 4C4G (已有) → 开发/测试环境
└── Arkclaw 2C4G (已有) → V1.0后独立跑MiroFish推演(可选)
```

### 方案对比

| 维度 | A: HK+MY双区 | B: 火山云单区 |
|------|-------------|-------------|
| LLM API延迟 | 30-80ms | 5-15ms |
| 爬虫覆盖 | 国内外30+平台 | 国内平台全覆盖 |
| 历史公众号源适配 | 知乎/B站/微博/豆瓣/微信 ✅ | 同左 ✅ |
| 境外学术源 | 可爬 | 受限（几乎不需要） |
| 架构复杂度 | 高（VPN+双区同步） | 低（单VPC内网） |
| 月费(中配) | ¥810-1,310 | ¥500-800 |
| ICP备案 | 不需要（域名需备案） | 需要（1-3周） |
| 推荐场景 | 需要境外数据源 | **历史公众号内容** ★ |

## 3. 模块复用策略

### 3.1 直接复用 BettaFish

| 模块 | 复用方式 | 改动量 |
|------|---------|--------|
| MindSpider | 增加历史平台爬虫适配器 | +5个平台适配器 |
| ForumEngine | 替换Agent提示词为历史7维 | 仅改提示词 |
| ReportEngine | 新增历史公众号模板 | +3个MD模板 |
| QueryEngine | 保持搜索能力，增加历史源配置 | 仅改配置 |
| InsightEngine | 接入历史数据库 | 仅改配置 |

### 3.2 简化集成 MiroFish

- 使用 MiroFish 的 OASIS 模拟引擎
- 限制 Agent 数量 50-100（vs 完整版数千）
- 模拟轮次 20-30 轮（vs 完整版数百轮）
- 仅用于"历史如果"选题场景

## 4. 管道接口设计

### 4.1 输出 API

```
POST /api/generate/article
{
  "topic": "安史之乱",
  "platforms": ["wechat_mp", "video_channel"],
  "depth": "deep",          // quick/standard/deep
  "enable_prediction": false // 是否启用MiroFish推演
}

Response:
{
  "article_id": "...",
  "wechat_mp": {
    "title": "...",
    "sections": [...],
    "markdown_content": "..."
  },
  "video_channel": {
    "script": [...],
    "duration_estimate": 180
  },
  "angles": [...],
  "references": [...]
}
```

### 4.2 Webhook 触发

```
POST /webhook/trigger/daily    # 触发日扫描
POST /webhook/trigger/weekly   # 触发周汇总
POST /webhook/trigger/monthly  # 触发月前瞻
```

### 4.3 对接现有 Python 管道

```python
# 现有管道中调用
import requests

def get_content(topic: str) -> dict:
    resp = requests.post(
        "http://hk-node:5000/api/generate/article",
        json={"topic": topic, "platforms": ["wechat_mp"]}
    )
    return resp.json()

# 返回的 dict 可直接注入现有编辑器/发布流程
```

## 5. 技术选型总结

| 层级 | 技术 | 理由 |
|------|------|------|
| LLM | 通义千问 qwen-plus | 主力模型，性价比高 |
| LLM (辅助) | DeepSeek | 长文本任务，成本低 |
| 搜索API | 安思派(Anspire) | BettaFish推荐，国内可用 |
| 数据库 | PostgreSQL | BettaFish原生支持 |
| 容器化 | Docker Compose | 已有Dockerfile |
| 前端 | Flask (BettaFish自带) | 运营看板 |
| 任务调度 | APScheduler | Python生态，轻量 |
| 记忆系统 | Zep Cloud | MiroFish依赖 |
